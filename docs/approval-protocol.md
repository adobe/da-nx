# DA Agent Approval Protocol (v2)

> **Status:** APPROVED 2026-07-24 — ready to implement (see §12 for resolved decisions).
> **Owners:** da-nx (client) + da-agent (server). This is *our* protocol; the
> front-end implements it with no external library dependency. The server keeps
> its internal use of the Vercel AI SDK, but the client↔agent wire contract
> below is ours and does **not** mirror the SDK's native approval format.

This replaces the organically-grown approval flow (separate
`tool-approval-request` / `tool-approval-response` parts, per-approval POSTs,
positional stale detection, orphan-stripping). Those are removed.

---

## 1. Goals

- A single agent step may gate **multiple** destructive tools. All of their
  approvals are surfaced and answered **together, in one round**.
- The protocol is **declarative and idempotent**: the server derives what to do
  by reconciling state keyed on `toolCallId`, never by message position.
- Remove concepts that only existed to patch the old design: `approvalId`
  pairing, the `role: 'tool'` message, `tool-approval-request` /
  `tool-approval-response` parts, positional "stale vs fresh" detection, and
  client-side orphan-stripping.

## 2. Non-goals

- Changing which tools require approval (still the 5 content tools +
  fail-closed MCP gating).
- Changing auth, page context, attachments, skills, or MCP payload fields.
- Adopting the AI SDK's UIMessage/approval shapes on the wire.

---

## 3. Concepts (glossary)

- **Conversation** — the entire chat for one `org--site--userId`, persisted in
  IndexedDB. An ordered list of **Messages**.

- **Message** — `{ role, content }` where `role ∈ { 'user', 'assistant' }`.
  **There is no `'tool'` role.** `content` is an ordered array of **Parts**.

- **Part** — a discriminated union (`type` field). Two kinds:
  - **Text part** — `{ type: 'text', text }`.
  - **Tool part** — one tool invocation and its whole lifecycle (see §4). This
    is the only place tool activity lives; results and decisions attach here.

- **Tool invocation** — a single call of one tool, identified by a
  globally-unique **`toolCallId`** generated by the agent. `toolCallId` is the
  **primary key** for everything: the UI card, the decision, and the result all
  reference it. (No `approvalId` exists.)

- **Turn** — one user request and all the work to satisfy it. Begins with a
  `user` message; ends when the agent yields control with nothing pending
  (final assistant text). A turn may pause for approval one or more times
  without ending.

- **Round** — one `POST /chat` request and its streamed response. A turn spans
  **one round if no approval is needed**, or **multiple rounds** when the agent
  pauses for approvals. Each round re-sends the full (reconciled) history; the
  agent is stateless per round.

- **Approval gate** — a round ends in the `awaiting-approval` state when the
  agent emitted one or more tool parts in state `awaiting-approval` and did
  **not** execute them.

- **Decision** — the user's `approve` / `reject` for one tool part. The client
  records a decision by transitioning that part's `state` (§4). Decisions for a
  round are answered **as a batch** (§6).

- **Approval queue** — when a round pauses with multiple `awaiting-approval`
  parts, the client presents them **one at a time as a queue** (not all at
  once), in the order the agent emitted them. The user decides the head of the
  queue, then the next, and so on. The batched POST is sent only after the
  **whole queue is drained** (§6).

- **Auto-approve ("always approve")** — a **client-only** convenience: choosing
  "always approve" on a queued item both (a) approves that item and (b)
  **immediately drains every remaining queued item of the same `toolName`** to
  `approved` without further prompting, and marks that `toolName`
  auto-approved so future `awaiting-approval` parts of it are approved on
  arrival. Invisible to the wire — the server only ever sees ordinary
  `approved` parts.

## 4. Tool-part lifecycle (state machine)

A tool part:

```jsonc
{
  "type": "tool",
  "toolCallId": "tc_abc123",     // primary key, agent-generated, stable
  "toolName": "content_update",  // ALWAYS present (contract fix — never omitted)
  "input": { /* tool args; may contain _da* client-only hints */ },
  "state": "awaiting-approval",
  "output": { "type": "json", "value": { /* … */ } }  // only when output-available
  // "errorText": "…"                                  // only when output-error
}
```

`state` values and transitions:

```
                        (approvalRequired=false)
   input-available ─────────────────────────────► output-available / output-error
        ▲                                              ▲
        │ agent emits (no approval needed)             │ server executed it
        │                                              │
   [agent step]                                        │
        │                                              │
        ▼ agent emits (approval needed)                │
   awaiting-approval ──approve──► approved ────────────┘
        │                                    (server executes; result streamed back)
        └───────────reject───────► rejected ──────────► output-error
                                            (server writes rejection result)
```

| state | set by | meaning |
|---|---|---|
| `input-available` | agent | tool call complete, **no approval** required; server will execute it |
| `awaiting-approval` | agent | needs a user decision; server did **not** execute |
| `approved` | client | user approved; **not yet executed** |
| `rejected` | client | user rejected; will not execute |
| `output-available` | server | executed successfully; `output` populated |
| `output-error` | server | execution failed **or** was rejected; `errorText` populated |

**Invariant the server reconciles on (idempotent, position-independent):**
- `approved` **and no** `output`/`errorText` → **execute now**.
- `rejected` **and no** `output`/`errorText` → **write rejection result** (no execution).
- `output-available` / `output-error` → **already done, do nothing**.

Because completion is recorded *on the same part* as `output-available`, a
replayed / reordered / retried history can never double-count or skip — this is
what removes Bug 4 structurally (no positional "stale vs fresh" test).

**Output shape:** `{ type: 'text' | 'json' | 'error-text', value }` (unchanged
from today's normalization).

## 5. Streaming events (agent → client)

Newline-delimited `data:` JSON events (as today). Event set:

| `type` | fields | client action |
|---|---|---|
| `text-delta` | `text` | append to current assistant text part |
| `tool-input-available` | `toolCallId, toolName, input, approvalRequired` | create/replace tool part; state = `awaiting-approval` if `approvalRequired` else `input-available` |
| `tool-output-available` | `toolCallId, output` | find part by `toolCallId`, set state `output-available` + `output` |
| `tool-output-error` | `toolCallId, errorText` | find part by `toolCallId`, set state `output-error` + `errorText` |
| `finish` | `reason` | end of round (see below) |
| `error` | `message` | surface error, end round |

`finish.reason`:
- `stop` — turn complete (assistant produced its final answer).
- `awaiting-approval` — round paused; ≥1 tool part is `awaiting-approval`. The
  client collects **all** such parts and prompts (§6).
- `length` / `error` — abnormal termination.

`toolName` is **required** on `tool-input-available` (kills the current
"recover toolName from the prior card" hack).

Optional (may add if useful, not required): `tool-input-start` /
`tool-input-delta` for streaming args into the card before completion.

---

## 6. Sending decisions (client → agent) — the batch rule

When a round finishes with `reason: 'awaiting-approval'`:

1. The client collects **every** tool part currently in `awaiting-approval`
   into a **queue**, in emission order.
2. It presents the queue **one item at a time** (see task #3). For each item the
   user approves or rejects it; the client transitions that part's `state` to
   `approved` / `rejected` **in place** and advances to the next item.
   - Choosing **"always approve"** on an item drains all remaining queued items
     of the same `toolName` to `approved` at once (§3), and auto-approves that
     `toolName` for the rest of the conversation.
3. When the queue is fully drained (every part decided), the client sends
   **one** `POST /chat` with the full reconciled history.

There is **no** separate decision message and **no** per-approval POST. The
decision *is* the mutated `state` on the existing assistant tool parts. This is
the core fix for Bug 1 (server now sees, and must process, all of them) and for
the "one-at-a-time" client anti-pattern.

**Reject:** rejecting still requires a POST if any sibling in the same batch was
approved (the agent must continue). If **every** part in the batch is rejected,
the client still sends one POST so the agent can react to the rejection(s) and
close out the turn gracefully (replaces today's "reject just stops locally").

## 7. Server round algorithm

On each `POST /chat`:

1. Parse + validate body (schema unchanged apart from message shape).
2. **Reconcile** the history (replaces `resolveApprovals`):
   - For every tool part with `state: 'approved'` and no result →
     collect for execution.
   - For every tool part with `state: 'rejected'` and no result →
     synthesize an `output-error` result `{ value: 'Action rejected by user.' }`.
   - Execute the collected approved tools **sequentially**, in emission order,
     stripping `_da*` client-only keys from `input` first. Map each result to
     `output-available` / `output-error`. Sequential is deliberate: these are
     destructive content ops against da-admin/collab, approvals are human-gated
     (so execution latency is irrelevant), and one-at-a-time gives deterministic
     ordering and avoids races on the same path/room. (This is why "Bug 3 —
     serial execution" is treated as a non-issue rather than parallelized; the
     real fix is executing *all* approved tools, not the order.) Note: non-gated
     reads the model fires within a step still run inside the SDK's own tool loop
     — this sequential rule governs the approved *destructive* batch we control.
   - Stream `tool-output-available` / `tool-output-error` back for each so the
     client updates its parts.
3. Build model messages for the LLM by expanding tool parts into the provider's
   `tool_use` + `tool_result` blocks (server-internal; the client never sees
   provider shapes).
4. Call `streamText` (SDK) to continue the turn. The LLM either:
   - produces final text → stream `text-delta` … → `finish: stop`; or
   - calls more tools — gated ones are emitted as `tool-input-available`
     (`approvalRequired: true`) and **not** executed → `finish: awaiting-approval`;
     non-gated ones execute and stream `tool-output-available`.
5. **Safety net (keep):** before handing messages to the provider, run the
   equivalent of `ensureOrphanedToolResults` so a step-limit cutoff
   (`stepCountIs(5)`) can never leave a `tool_use` without a `tool_result`
   (independently useful; see PR #36). This is the only thing standing in for
   Bug 2 once Bug 1 is fixed.

Deleted server pieces: `tool-approval.ts` as an *approval* module (the
continuation-stream helper is folded into step 2's streaming);
`resolveApprovals`'s positional stale/fresh branch; the `hasPendingApprovals`
early-return in `server.ts`.

`needsApproval: async () => true` stays on the 5 content tools, and the
fail-closed MCP gating stays — that's how the agent decides to emit
`approvalRequired: true`.

## 8. Removed concepts and why

| Removed | Why it's safe to remove |
|---|---|
| `approvalId` | `toolCallId` is a sufficient, already-unique key for request↔decision↔result. |
| `role: 'tool'` messages | Tool results live on the assistant tool part; server expands to provider shapes internally. |
| `tool-approval-request` / `tool-approval-response` parts | Superseded by the tool-part `state` machine. |
| positional "stale vs fresh" (`lastConversationIdx`) | Reconciliation keys on `state`+`toolCallId`; order no longer matters. **Kills Bug 4.** |
| client `stripOrphanedToolCallMessages` | No orphan tool-role messages can exist; the server's `ensureOrphanedToolResults` remains as the provider-side guard. |
| `_onToolEvent` message-promotion into synthetic assistant messages | Tool parts are created directly from `tool-input-available` events. |

## 9. Persistence & migration

- IndexedDB store gets a **schema version bump**.
- On load, histories written in the **v1** shape are **best-effort migrated**:
  text parts pass through; any resolved v1 tool activity
  (`tool-call` + `tool-result`) maps to a `tool` part with
  `state: output-available`; any *unresolved* v1 approval artifacts are dropped
  (they were, by definition, mid-flight and are the buggy state we're removing).
- If a stored conversation can't be cleanly migrated, keep its **text** turns
  and drop tool cruft — never hard-fail the chat load.

## 10. Worked example — two gated tools in one step

**Round 1** — user asks to update two files.

Client → agent: `[ { role: user, content: [{type:text, text:"update A and B"}] } ]`

Agent → client stream:
```
tool-input-available {toolCallId: t1, toolName: content_update, input:{path:"/A"}, approvalRequired:true}
tool-input-available {toolCallId: t2, toolName: content_update, input:{path:"/B"}, approvalRequired:true}
finish {reason: "awaiting-approval"}
```
Client shows both approval cards (t1, t2).

**Round 2** — user approves t1, rejects t2. Client mutates parts and POSTs the
full history; the assistant message now contains:
```
{type:tool, toolCallId:t1, toolName:content_update, input:{path:"/A"}, state:"approved"}
{type:tool, toolCallId:t2, toolName:content_update, input:{path:"/B"}, state:"rejected"}
```
Agent reconciles: executes t1 (the only approved item), writes rejection for
t2, streams:
```
tool-output-available {toolCallId: t1, output:{type:json, value:{...}}}
tool-output-error     {toolCallId: t2, errorText:"Action rejected by user."}
text-delta {text:"Updated A. Skipped B (rejected)."}
finish {reason:"stop"}
```
Turn complete. Both decisions honored (Bug 1 fixed); execution was sequential
and deterministic (Bug 3 intentionally not parallelized); no positional logic
anywhere (Bug 4 gone); no orphaned `tool_use` (Bug 2 handled by the safety net).

| Bug | Fixed by |
|---|---|
| **1** only first approval processed | §6 batch + §7 reconcile *all* `approved` parts |
| **2** orphaned tool-calls cascade | falls out of Bug 1; `ensureOrphanedToolResults` retained (§7.5) |
| **3** serial execution | intentionally kept sequential (§7) — a non-issue under human-gated approval; the real fix is executing *all* approved tools (Bug 1), not the order |
| **4** fragile stale/fresh | designed out — §4 idempotent reconciliation on `state`+`toolCallId` |

## 12. Decisions (RESOLVED 2026-07-24)

1. **Naming** — ✅ keep `content` (array of parts); part `type: 'tool'`.
2. **Cross-round re-execution of non-idempotent tools** — ✅ **accept the risk
   now** (matches today's behavior). Server-side dedupe by `toolCallId`
   (collab room / KV keyed on `sessionId+toolCallId`) is a **fast-follow**, not
   part of this change. If the interruption window is hit, an approved tool may
   run twice; acceptable for v2.
3. **Reject-all still POSTs** — ✅ yes; the agent gets a chance to respond to a
   full rejection rather than the client silently stopping.
4. **Message boundaries** — ✅ yes; multiple assistant messages per round are
   allowed (parts are found by `toolCallId` globally, so boundaries don't affect
   correctness).
